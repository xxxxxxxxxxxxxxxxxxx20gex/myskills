# Layout Decision Table

Use this file when the source content can be presented in multiple ways. Choose the structure that best proves the slide's core message.

## Content Signals

| If the content contains | Use | Avoid |
|---|---|---|
| Sources, processing, calls, outputs, review, accountability | Evidence chain, audit panel, traceability loop | Generic four cards |
| Modules, dependencies, service layers, platform components | Layered architecture, hub-and-spoke map, system block diagram | Timeline |
| Inputs, stages, algorithms, outputs | Data pipeline, model pipeline, numbered process flow | Capability grid |
| Baseline, proposed method, competitors, alternatives | Comparison matrix, before/after split, delta cards | Process flow |
| Metrics, results, evaluation, quality indicators | Metric dashboard, chart-first layout, ranked evidence panel | Paragraph explanation |
| Phases, milestones, deliverables, schedule | Timeline, phase lanes, roadmap | Architecture diagram |
| Risks, controls, safety checks, exceptions, audit | Risk-control loop, governance matrix, review checklist | Decorative warning icons |
| User roles, collaboration, permissions, handoffs | Swimlane, role-process matrix, responsibility map | Unlabeled node graph |
| Before state, after state, transformation | Before/after contrast, maturity ladder, upgrade path | Dense table |
| Capabilities, functions, features | Capability grid grouped by workflow or subsystem | Ungrouped bullet list |

## Selection Rules

- If the page needs to answer "how does it work", use a flow, pipeline, or architecture.
- If the page needs to answer "why is it better", use a comparison, proof panel, or metric dashboard.
- If the page needs to answer "why is it trustworthy", use an audit loop, evidence chain, or control matrix.
- If the page needs to answer "what can it do", use a capability grid grouped by task, role, or subsystem.
- If the page needs to answer "how will we build it", use a timeline, technical route, or phase lane.
- If the page contains both process and proof, make the process the main visual and put proof in a compact side panel.

## Fallback

When the content is too vague, do not invent a complex architecture. Use a two-column structure:

- Left: problem, method, or pipeline sketch.
- Right: structured details, assumptions, evidence, or open items.
- Bottom: one sentence stating the value, risk boundary, or next action.
